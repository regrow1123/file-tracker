## 증분 스냅샷 백업 시스템 설계 결정

### 아키텍처
- 노드(file-tracker) → Kafka → 소비자(backup-consumer) → restic → MinIO(S3)
- Kafka = 디커플링 버퍼. 소비자 추가만으로 용도 확장 가능
- DB 직접 넣기 X → 수백 노드 동시 write 시 DB 병목

### 변경 DB
- Kafka 이벤트를 변경 DB에 upsert → 중복 제거 (같은 파일 100번 수정 → 1건만 백업)
- DB 선택: Redis (소비자 여러 대 가능) 또는 SQLite (단일 소비자)
- 키: `(hostname, path)`, 값: 최신 이벤트 타입

### 백업 도구: restic
- BSD 2-Clause, 상업적 사용 무료
- 청크 기반 dedup → 1GB 파일 10버전 ≈ 1GB+α (S3 versioning은 10GB)
- S3 네이티브 지원 (`restic -r s3:http://minio:9000/bucket`)
- `--files-from` 으로 변경 파일만 선택 백업
- 대안 검토: borg(SSH만, S3 X), kopia(OK, 병렬 강력), duplicacy(상용 유료)

### 스토리지: MinIO
- S3 호환 오브젝트 스토리지 (자체 구축)
- restic 백엔드로 사용
- S3 versioning은 전체 복사본이라 비효율 → restic 청크 dedup 사용

### 원본 파일시스템: Lustre
- 수PB 규모, 수백억 파일

### Kafka 개념
- offset commit = "여기까지 처리 완료" 마커. 소비자 재시작 시 이어서 읽기
- 메시지 소비해도 삭제 안 됨. retention 설정(기본 7일)에 따라 자동 삭제
- 같은 메시지를 여러 소비자가 독립적으로 읽기 가능
